Contact / public MVP

정정, 소스 제안, 협업 문의는 지금 단계에서는 GitHub를 공식 창구로 받습니다

아직 전용 inbox를 열지 않았기 때문에, 현재는 GitHub issue가 가장 투명한 public contact channel입니다. 중요한 건 채널보다 패킷 품질입니다. 공식 링크와 영향 범위가 함께 와야 실제 큐로 들어갈 수 있습니다.

Send the smallest useful packet

Contact rule
Required

공식 링크와 왜 중요한지, 누가 영향받는지까지 함께 보내야 큐에 들어갑니다

“이거 봐주세요”보다 작은 실행 패킷이 더 유용합니다. source, impact, audience, action을 같이 보내는 편이 낫습니다.

Correction wrong fact or stale link
Source tip official update worth briefing
Collaboration distribution or ops workflow fit

Current limitation

  • 전용 이메일 inbox는 아직 열려 있지 않습니다.
  • 개인정보 수집형 폼도 아직 운영하지 않습니다.
  • 공개 트레일이 필요한 이유로 GitHub를 우선 사용합니다.

What to send

문의 유형마다 필요한 정보가 다릅니다

channel보다 중요한 건 패킷의 선명도입니다. 아래 항목 정도만 있으면 수동 triage가 훨씬 빨라집니다.

Lane 01

Correction request

사실 오류, 끊어진 링크, 잘못된 해석을 바로잡는 요청입니다.

  • 문제가 있는 페이지 URL
  • 정정이 필요한 문장 또는 블록
  • 수정 근거가 되는 official source

Lane 02

Source tip

새 seed 후보가 될 만한 공식 업데이트를 제안하는 경우입니다.

  • official link
  • 왜 실무 영향이 큰지
  • 누가 가장 먼저 봐야 하는지

Lane 03

Collaboration note

이 포맷이 특정 팀이나 제품군에 더 맞는지, 어떤 distribution surface가 필요한지에 대한 제안입니다.

  • 추적 중인 제품 또는 카테고리
  • 현재 운영 워크플로우의 막힘
  • 원하는 output format

Current channels

현재 열려 있는 public channel

Open an issue 정정 요청, 소스 제안, beta dispatch 문의의 기본 창구 Review existing issues 이미 열려 있는 요청과 중복 여부를 먼저 확인 Repository 현재 사이트 구조와 PR 흐름을 직접 확인할 수 있는 기준점

What happens next

보내주신 내용은 이런 순서로 처리됩니다

1. verify source official authority가 있는지 먼저 확인합니다.
2. decide queue brief, weekly follow-up, or backlog인지 분류합니다.
3. fold into output seed brief, weekly desk, checklist, newsletter pack 중 하나로 연결합니다.

Public feedback loop

연락은 결국 다음 brief와 weekly queue를 더 정확하게 만드는 데 쓰입니다

어떤 채널을 더 원하시는지보다 어떤 packet을 받고 싶은지가 더 중요합니다. ops용인지, eng용인지까지 함께 알려주세요.
